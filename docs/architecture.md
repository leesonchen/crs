# Claude Relay Service - 系统架构文档

## 1. 系统整体架构

### 1.1 核心设计理念

Claude Relay Service 采用**分层架构设计**，实现高度模块化和可维护性。最新架构重构专注于**双方向桥接模式**和**统一调度器**，支持 Claude CLI 与 OpenAI API 之间的无缝转换。

### 1.3 分层架构详解

#### 路由层 (Routes)
- **API路由**: `/api/v1/*` - Claude API兼容端点
- **OpenAI路由**: `/openai/*` - OpenAI兼容端点
- **管理路由**: `/admin/*` - Web管理界面API
- **健康检查**: `/health` - 系统状态监控

#### 服务层 (Services)
- **统一调度器**: 跨平台账户选择和负载均衡
- **桥接服务**: 双向API格式转换（Claude ↔ OpenAI）
- **中继服务**: 请求转发和响应处理
- **账户服务**: 多平台账户管理和OAuth处理

#### 数据层 (Models)
- **Redis集群**: 高性能数据存储和缓存
- **文件系统**: 配置、日志、静态资源
- **外部API**: 上游AI服务提供商

## 2. 桥接模式架构详解

### 2.1 双向桥接设计

桥接模式是最新架构重构的核心，实现了 **Claude API ↔ OpenAI API** 的双向格式转换：

#### 桥接方向说明

1. **Claude → OpenAI**: Claude Code CLI 请求 OpenAI 模型
   - 输入：Claude API 格式 (`/api/v1/messages`)
   - 输出：转换为 OpenAI Responses 格式，转发到 OpenAI 账户
   - 响应：OpenAI 响应转换回 Claude 格式返回给客户端

2. **OpenAI → Claude**: Codex CLI 请求 Claude 模型
   - 输入：OpenAI Responses 格式 (`/openai/responses`)
   - 输出：转换为 Claude API 格式，转发到 Claude 账户
   - 响应：Claude 响应转换回 OpenAI 格式返回给客户端

### 2.2 简化的双层映射架构

```javascript
// 系统级桥接配置 (Redis: system:bridge_config)
{
  "openaiToClaude": {
    "enabled": true,
    "defaultModel": "claude-3-5-sonnet-20241022",
    "modelMapping": {
      "gpt-5": "claude-sonnet-4-20250514",
      "gpt-5-mini": "claude-3-5-haiku-20241022"
    }
  },
  "claudeToOpenai": {
    "enabled": true,
    "defaultModel": "gpt-5",
    "modelMapping": {
      "claude-sonnet-4-20250514": "gpt-5",
      "claude-3-5-haiku-20241022": "gpt-5-mini"
    }
  }
}
```

**映射优先级**：
1. **系统级配置 (Layer 1)**: 控制跨平台虚拟模型到目标平台模型的映射
2. **账户级适配 (Layer 3)**: 同平台内的模型能力适配，确保账户支持映射的模型

**移除 Layer 2**: 避免过度复杂化，多租户通过 API Keys 绑定特定账户/分组来管理

### 2.3 桥接模式数据流

**修正后的架构流程**: 用户请求 → 路由层 → 调度器(选择账户) → 桥接判断 → [桥接服务] → 中继服务 → 上游 API

## 3. 统一调度器架构

### 3.1 跨平台账户管理

统一调度器是系统的核心组件，**专注于账户选择和负载均衡**，不再负责模型映射和格式转换：

**修正后的核心职责**：
- **账户选择**: 根据请求模型和会话粘性选择最合适的账户
- **桥接模式激活**: 当原生平台账户不可用时，自动启用桥接账户
- **负载均衡**: 在多个可用账户间进行智能负载分配
- **健康检查**: 监控账户状态和可用性

#### 支持的账户类型
- **Claude OAuth**: 通过 OAuth 2.0 授权的 Claude 账户
- **Claude Console**: Claude Console 直接访问账户
- **Bedrock**: AWS Bedrock 账户
- **Gemini**: Google Gemini API 账户
- **OpenAI**: OpenAI API 账户（支持桥接）
- **OpenAI Responses**: OpenAI Responses API 账户（支持桥接）

│   ├── usage:cost:daily:{keyId}:{date}   → String (每日费用)
│   ├── usage:cost:total:{keyId}          → String (总费用)

├── 账户存储区

    ├── temp:{key}                         → String (临时数据)

## 7. 总结

这个架构设计充分考虑了：

1. **高可用性**: 多层负载均衡、故障转移、健康检查
2. **高性能**: 缓存优化、异步处理、批量操作
3. **可扩展性**: 模块化设计、插件机制、服务接口
4. **可维护性**: 清晰的分层、完善的日志、监控告警
5. **安全性**: 多重认证、访问控制、数据加密

整个系统能够支持从单实例部署到大规模集群部署的平滑扩展，满足不同规模用户的需求。
